# skill: gh-pages

Always use the \`gh-pages\` branch directly when serving static content (like mockups) via GitHub Pages if the repository is configured to build from the \`gh-pages\` branch directly instead of an Actions workflow.

When pushing standalone static page mockups to GitHub pages:
1. Verify what branch GitHub Pages is sourcing from (usually \`gh-pages\`).
2. Make your edits in \`main\`.
3. Test locally.
4. When ready to publish: do NOT commit direct changes to \`main\` and try to merge unrelated histories.
5. Use \`git checkout gh-pages\` -> \`git checkout main -- path/to/file.html\` -> \`git commit\` -> \`git push origin gh-pages\`.

## Workaround for Unrelated Histories
If you pull down a vanilla template or run an npm script that forcibly rewrites the \`gh-pages\` branch (like gh-pages cli module), your local \`gh-pages\` branch might be detached or have an unrelated history to \`main\`. 

Never do a force push (\`-f\`) without checking if the commit graph is shared, because doing so overwrites the deployment structure that build tools naturally generate.

## Fast Sync Template
If you ever just need to copy files identically from \`main\` to \`gh-pages\`:

\`\`\`bash
git fetch origin gh-pages
git branch -D gh-pages
git checkout -b gh-pages origin/gh-pages
git checkout main -- path/to/files/
git commit -m "chore: manual sync from main"
git push origin gh-pages
git checkout main
\`\`\`

### Handling Auto-Deploy Overwrites
If you are modifying static HTML files generated by your root build process, make sure your modifications exist on the source branch (typically \`main\`) **before** the build pipeline triggers. If you only modify files on the target \`gh-pages\` output branch, the next pipeline run will overwrite your changes. Always edit source static assets on the source branch.
If a Github action runs `npm run deploy`, you must let the deploy script finish *before* you attempt a `git checkout gh-pages` update, or they will collide.
